Common Events
Slave Maker 3 SDK Version 8

Basic Concepts
The game works on an event/handler model, where you start an event by showing some text and images and then show a button or question. This button has a linked eventid that uniquely defines the event.

When the button is clicked then an event handler function is called by the core game, and it is called using actionscript and xml interfaces for
  • the currently trained slave
  • for all minor slaves (xml interface only)
  • the current assistant
  • all custom events like the SMEncounter event, or the Lady Grey event
Each of these will check if they can handle the event and if so will process, generally clearing text, images and showing new text/images, plus optionally showing a new button to continue the event chain. For most of the handlers, they return true to indicate the event was handled.

By default when your slave is training (ie performing acts like chores, jobs, sec acts etc) the Planning Next button is always shown. During early morning and daily events the Next button is shown by default.

Generally an event works as follows
  • Call a button show function which sets the event variables (see below)
  • the player clicks the button
  • the button Handler is called to process the event
  • a Post Handler is called after the Handler finishes processing
In the Handler or Post Handler a new event can be started as desired by the developer.


Eventid
An eventid is either a number, like 300, or a string like "RanmaGenderChange". The game reserves some numbers for slave, assistant, event use
300-999  - reserved for slave girl use
7000-7999        - reserved for assistant use
10000-10999      - reserved for custom event use

For string eventid's they must be a unique name, generally recommend prefixing it with the name of the slave or event, thus
"RanmaGenderChange"
"LadyGreyOnsen"
etc
If you do not use a unique name then you will disable the existing event, but it depends where it is defined, see the sequence of calling above, so an assistant version would never happen if it conflicts with a slaves. This is the same for the event numbers.

For most events the variables
_root.NumEvent
_root.StrEvent
hold the eventid. NumEvent is a numeric event, and StrEvent is a string event. If StrEvent has a value then it is triggered, otherwise NumEvent will be triggered. This is an unusual case and seldom used in the game, mainly for some complex gender based cases and for your slave to override and bypass the standard event.

When event handler function is called the following variables are also set
_root.OldNumEvent
_root.OldStrEvent
these are set to the eventid when the event starts. This is to allow Post Handlers to determine the original event. This because an event can show a new event to trigger a chain of events.

Buttons
See Core - Buttons.html for details on the buttons that can be shown for an event. These include

DoEvent - do an event and show the NextEvent button.
DoYesNoEvent - display yes/no buttons at the bottom of the general text area
AskHerQuestions - ask up to 5 questions
ResetQuestions - Reset and hide all questions
AddQuestion - Add a question to the list, call 1+ times.

Core Variables

When an event is called or set the game uses a standard set of variables to record what event is to be done or is being done. A separate set of variables also holds what event was last done.

Actionscript Variable XML Variable

NumEvent

 currentnumevent (READ ONLY)
(Number) The numeric eventid for the current event or the event to be next performed. Generally this is set by the various button functions like DoEvent() and tested in the button/event handler like DoEventNext().

eg
actionscript: xml:
function DoEventNext() {
   if (_root.NumEvent == 300) {
      // doing event 300
      // do whatever here
     return true;
   }
   return false;
}
 <if currentnumevent='300'>
   <!-- doing event 300 -->
</if>

StrEvent

currentstrevent
(String) The current string eventid, otherwise identical to NumEvent.

OldNumEvent

lastnumevent
(Number) The value of NumEvent at the start of an event. Useful to find out the event number before the event itself changes or starts a new event.

DoneEvent

 no xml implementation
(Number) This is a flag to indicate if an event has been started. Any of the functions like DoEvent(), AskQuestions() etc will set this to a value 1

Primarily this flag is used in Events() below. Every test for an event there should first test to see if DoneEvent is 0 to prevent an event being overwritten/relpaced. You do not need to check this for PreEvent() below.

Event Pickers

There are a set of events where the game can choose one of a list of possible events. The implementation of these is quite different between xml and actionscript.

Early Morning Events

Each morning at 6am an event can happen. There are no standard events that happen here, all are custom for the slave. This event can be a series of events if desired.

If an event is triggered then the game by default will show a Next button that will perform the event 250. The is the same as the game automatically doing
_root.DoEvent(250);
But, this Next button is only shown if you did not explicitly show a button or question in the PreEvent() function.

By default, once these events are complete the game advances time by 1 hour, standardly to 7am. Any event triggered can advance time arbitrarily, see Core - Date, Time & the Diary.html.

XML:

<Events><EarlyMorning>

<Events>
     <EarlyMorning>
           <MyEvent1 conditionals>
            </MyEvent1>
   </EarlyMorning>
</Events>

The <EarlyMorning> node will contain a list of unique events, each with a conditional to determine if it should happen. The game sequentially checks each event until one's conditionals evaluate to true, then that event happens.

Note: if a node does not have any conditionals then it will never be selected. You can do this so the event node can be called from another node, say as a question response.

XML:

<Events><Daily>

<Events>
     <Daily>
           <MyEvent1 conditionals>
            </MyEvent1>
   </Daily>
</Events>

The <Daily> node will contain a list of unique events, each with a conditional to determine if it should happen. The game sequentially checks each event until one's conditionals evaluate to true, then that event happens. By default the game will show the Next button and schedule the eventid 9999 (see above). Also the DoneEvent test is not needed, this node is not checked if another event has been selected.

Note: if a node does not have any conditionals then it will never be selected. You can do this so the event node can be called from another node, say as a question response.

Example:
(from the core game Events.xml, slightly modified) This one asks a question, this the event 9999 is not scheduled.
