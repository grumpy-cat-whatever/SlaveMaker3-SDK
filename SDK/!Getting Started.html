Basic Concepts

The game works on this structure

a) Core game engine
handles all game-play and common events and images. This is contained mainly in SlaveMaker3.swf (or SlaveMaker3.exe) but also others like Tentacles.swf, Contests.swf. These additional files are discrete parts of the game, separated to simplify game development but mainly to lessen resource use when programming, I have reach times with 4+Gb of memory being used Additionally there are images in the subfolder Images and xml code in the Languages subfolder.

The core game contains many support functions that can be used by developers using actionscript or the built-in xml language.

b) SlaveGirl files
All slavegirl specific images and code is present in a set of files. There are two primary types of slaves, actionscript slaves and xml slaves.

actionscript:
  • an image to be shown in the slave market, must be named Slave-slavename.png
  • a control text file SlaveGirlXX.txt that is used by the game to define if the slave is present
  • code and images are compiled into a swf file using Adobe Flash or FlashDevelop. This must be developed for Actionscript 2, an older version of the language. This language is close to java in syntax.
  • optionally you can also have separate images in jpg, png, gif, swf format. These can be dynamically loaded for a particular event or scene.
  • optionally an xml file named Slave-slavename.xml that can be used for initialisation, events or other. This file is easily translatable.
XML:
  • the image and control text for actionscript are also used. Some slightly different fields are required for the control text file
  • an xml file named Slave - slavename.xml that contains all initialisation and code for the slave.
  • a folder of images with a set structure of filenames for the various images the game requires.
Slave files are stored in the folder Slave. Any external images are stored in the folder
Images/Slaves/slavename

c) Assistant files
During training you pick an assistant to aid you. An assistant is either a pre-trained slave, or a simplified version of a slave, developed using actionscript or xml. A slave maker assistants files are named
Assistant-assistantname.swf
Assistant-assistantname.xml
There is no slave market image or SlaveGirlXX.txt file. These are stored in the folder Assistant.

For xml assistants their images are held in the folder
Images/Assistants/assistantname

d) Minor Slaves
Your slave maker can purchase slave to assist in training. They are generally lesser slaves with lower potential and limited training, often with a single good feature, for instance a cook, a nurse etc. These are developed similarly to xml slaves, and cannot be developed using actionscript. They require
  • an image to be shown in the slave market, must be named Slave-slavename.png
  • optionally 2 or more images when talking to the slave or having sex. More optional images for other events, notably love confessions and pregnancy.
  • some xml code similar to xml slaves.

e) Interface Functions
A slave interacts with the core game through a set of interface functions. During trainings or events the core game will call these functions. Both actionscript and xml use this logical process but differ in implementation.

actionscript:
For instance to show the dance school image the game calls
SlaveGirl.ShowSchoolDance()
where SlaveGirl is the general movieclip the slavegirl is loaded into.

Many slavegirl functions are optional, but many need to be defined, also referred here are overridden (they are not class members now, but I am a c++ programmer, and they may become members eventually)

Most commonly for a slave girl there are a series of movieclips, one for each sex act, school, event etc
In the function the movieclip is shown and an appropriate alternative is selected.
I generally make each clip with several frames, one per alternative and show say

function ShowSexActTitFuck()
{
         if (_root.DickgirlXF > 0) TitsFuckClip.gotoAndStop(2);
         else if (_root.Lesbian) TitsFuckClip.gotoAndStop(3);
         else TitsFuckClip.gotoAndStop(1);
         TitsFuckClip._visible = true;
}
Remember naked versions where possible

Please check the class SlaveModule for the full list of interface functions
Common Scripts/Classes/SlaveModule.as

XML:
In your slave's xml you implement the <Images> node to define the images automatically shown for an act or event. The game automatically picks a random image and shows standard text.

You can customise this by implementing nodes in <DoPlanning><Acts> to change text, select images and trigger events.

A number of events actually call specific nodes and execute the xml code that you optionally implement for them.

f) Screen and Text Areas
Images and text are shown in 4 areas

1) Main Window - the large window, upper left. It is 4x3 aspect ratio and is 448x336 pixels now

2) People Window - the smaller window on the lower left. It is square 1x1 aspect ratio, 153x153 pixels, and is used to show people, your assistant etc

3) General Text field - the central text area, about 18 lines

4) Larger Text field - the area that can replace the statistics box, as of version 16/Slave Maker 3 it is scrollable and unlimited in length

Slavegirl Override Functions

These functions an be defined in the Slave Girl actionscript and are called by the main engine when certain actions are taken. Most of these are duplicated in _root scope and call these. To prevent infinite loops do not call say _root.HideImages() from within a local override of HideImages()
